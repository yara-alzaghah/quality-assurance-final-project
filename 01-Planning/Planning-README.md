# Part 01 – Test Planning

## Overview
This folder contains the **test planning deliverables** for the SauceDemo web application, prepared as part of the Software Testing – Quality Assurance capstone assignment.

The purpose of this phase is to define the **testing strategy, scope, approach, and scenarios** before test execution begins, following industry-standard QA practices.

---

## Contents of This Folder

### 1. SauceDemo_Test_Plan.docx
This document defines the overall **Test Plan** for the SauceDemo web application.

It includes:
- Project overview and testing objectives
- Scope (in-scope and out-of-scope activities)
- Assumptions and dependencies
- Test approach, levels, and types
- Test design techniques
- Environment and tools
- Defect management and reporting strategy
- Metrics and reporting
- Risks and mitigation
- Entry and exit criteria
- Schedule and resources

The test plan aligns with the assignment requirement for **Part A – Planning (Manual)** and demonstrates structured QA planning.

---

### 2. Happy and Negative Path Scenarios.docx
This document contains the **high-level test scenarios** covering both happy and negative paths.

The scenarios:
- Are mapped to core SauceDemo features (login, products, cart, checkout, logout)
- Include at least 10 clear scenarios as required by the brief
- Cover positive flows and validation/error handling
- Serve as the foundation for detailed manual test cases in Part B

---

## System Under Test
- **Application:** SauceDemo (https://www.saucedemo.com)
- **Type:** Web-based e-commerce demo application
- **Focus Areas:** Authentication, product inventory, cart management, checkout, and logout

---

## Tools Used
- Documentation: Microsoft Word
- Browser: Google Chrome / Microsoft Edge
- Test Design Techniques: Black Box, Equivalence Partitioning, Boundary Value Analysis

---

## Relation to Assignment Brief
This folder satisfies **Part A – Planning (Manual)** requirements:
- Test Plan with scope, assumptions, risks, and entry/exit criteria
- Clearly defined happy and negative test scenarios
- Proper documentation structure suitable for professional QA work

---

## Notes
- No test execution is performed in this phase.
- These documents are used as input for **manual testing (Part B)** and **UI automation (Part E)**.
